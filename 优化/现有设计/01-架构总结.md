## 二、核心模块详解

### 2.1 AgentService - 会话管理层

**文件**: `src/main/agent.ts`

**职责**:

- 管理 Topic（话题）与 Host（主机）的关联
- 终端生命周期管理（创建、关闭、重命名）
- 用户消息入口处理
- 授权请求处理

**核心数据结构**:

```typescript
class AgentService {
  // 会话注册表: Map<topicId, Map<hostId, AgentSession[]>>
  private topicSessions: Map<string, Map<string, AgentSession[]>>

  // 授权请求等待队列
  private pendingRequests: Map<string, (response: AuthResponse) => void>
}

interface AgentSession extends TerminalSession {
  paused: boolean // 是否被用户暂停
}
```

**关键方法**:
| 方法 | 功能 | 调用场景 |
|------|------|---------|
| `createTerminal()` | 创建新终端 | Agent 需要新终端时 |
| `closeTerminal()` | 关闭终端 | 任务完成或用户关闭 |
| `ensureSession()` | 获取或创建会话 | 执行命令前确保有终端 |
| `handleMessage()` | 处理用户消息 | 用户发送消息时 |
| `handleAuthResponse()` | 处理授权响应 | 用户批准/拒绝命令时 |

### 2.2 AgentRunner - 执行引擎层

**文件**: `src/main/AgentRunner.ts`

**职责**:

- ReAct-Verify 模式执行循环
- 工具调用与结果处理
- 上下文组装与压缩
- 并行执行调度

**执行流程**:

```
用户消息 → run(history)
    │
    ▼
循环 (maxTurns = 30):
    │
    ├── 1. 组装上下文 (ContextAssembler)
    │      - System Prompt
    │      - 终端上下文 (buildTerminalContext)
    │      - 记忆召回 (MemoryManager.recallRelevantContext)
    │      - 历史消息
    │
    ├── 2. 调用 LLM (ProviderAdapter)
    │      - 获取 assistant message
    │      - 获取 tool calls (如果有)
    │
    ├── 3. 检查终止条件
    │      - 无 tool calls → 返回最终结果
    │
    ├── 4. 执行工具 (executeGrouped)
    │      - 同主机串行执行
    │      - 不同主机并行执行
    │      - DoomLoop 检测
    │
    ├── 5. 处理结果
    │      - 格式化观察 (formatObservation)
    │      - 大输出提纯 (MemoryManager.distillObservation)
    │      - 推入 turnMessages
    │
    └── 6. 检查上下文压缩
           - 超过 90% 阈值 → 自动压缩
```

**核心配置**:

```typescript
const MAX_AGENT_TURNS = 30 // 最大轮数
const AGENT_TEMPERATURE = 0.2 // 温度
const AUTO_COMPACT_THRESHOLD = 0.9 // 自动压缩阈值
const DOOM_LOOP_THRESHOLD = 5 // 死循环检测阈值
```

### 2.3 CommandExecutor - 终端执行层

**文件**: `src/main/terminal.ts`

**职责**:

- 管理所有终端会话
- 执行命令（Agent 或用户）
- 捕获输出
- 会话锁定机制

**核心数据结构**:

```typescript
class CommandExecutor {
  // 会话存储: Map<sessionId, Session>
  private sessions: Map<string, Session>

  // 会话状态
  interface Session {
    id: string
    hostId: string
    topicId: string
    stream: TerminalStream    // PTY/SSH 流
    buffer: string            // 输出缓冲区
    isLocked: boolean         // 是否被锁定
    lockedBy: 'agent' | 'user' | null
    lastCommand: string       // 最后命令
    lastOutput: string        // 最后输出
    shellIntegration: boolean // Shell 集成状态
  }
}
```

**双入口设计**:

```typescript
// Agent 入口
async executeAgentCommand(
  sessionId: string,
  command: string,
  options?: CommandOptions
): Promise<CommandResult>

// 用户入口
async handleUserInput(
  sessionId: string,
  input: string
): Promise<void>
```

**会话锁定机制**:

- Agent 执行命令时锁定会话 (`isLocked = true, lockedBy = 'agent'`)
- 锁定期间用户输入被缓冲或拒绝
- 支持 Ctrl+C 透传，用户可随时接管
- 命令完成后解锁

**OSC 信号追踪**:

- 通过 Shell 集成发送 OSC 序列
- 追踪命令开始、结束、退出码
- 支持命令执行时间统计

### 2.5 SubAgent 系统

**文件**: `src/main/agent/task-tool.ts`

**实现方式**:

- 通过 `task` 工具派生
- 独立 session ID: `sub_${agentName}_${uuid}`
- 隔离上下文执行
- Token 消耗汇总到父会话

**支持的子 Agent**:
| 子 Agent | 用途 | 模式 |
|----------|------|------|
| `explore` | 只读调查 | secondary |
| `verify` | 快速验证 | secondary |
| `build` | 主 Agent（不可作为子 Agent） | primary |

**调用示例**:

```typescript

  "tool": "task",
  "arguments": {
    "agent": "explore",

  }
}
```

### 3.2 事件总线机制

**文件**: `src/main/agent/event-bus.ts`

**发布的事件类型**:
| 事件名 | 触发时机 | 数据 |
|--------|---------|------|
| `agent:thinking` | Agent 开始思考 | topicId, taskId |
| `agent:step` | Agent 执行步骤 | topicId, taskId, stepId, role |
| `agent:tool-call` | 调用工具 | topicId, taskId, toolName, args |
| `agent:tool-result` | 工具返回结果 | topicId, taskId, toolName, output |
| `agent:task-complete` | 任务完成 | topicId, taskId, status, summary |
| `agent:doom-loop` | 检测到死循环 | topicId, taskId, toolName, callCount |
| `agent:auto-compact` | 上下文压缩 | topicId, taskId, originalTokens, compactedTokens |
| `agent:subagent-complete` | 子 Agent 完成 | topicId, taskId, subagentSessionId, usage |
| `agent:usage` | Token 使用统计 | topicId, taskId, inputTokens, outputTokens |

**使用方式**:

```typescript
// 发布事件

  topicId: 'topic-123',
  taskId: 'task-456',
  toolName: 'execute_command',
```

## 四、数据库存储结构

### 4.1 完整表结构

```sql
-- 1. 主机配置表
CREATE TABLE hosts (
  id TEXT PRIMARY KEY,           -- 主机唯一ID
  alias TEXT NOT NULL,           -- 显示名称
  ip TEXT NOT NULL,              -- IP 地址
  port INTEGER NOT NULL,         -- SSH 端口
  username TEXT NOT NULL,        -- 用户名
  password TEXT,                 -- 密码（加密存储）
  keyPath TEXT,                  -- SSH key 路径
  tags TEXT,                     -- 标签（JSON 数组）
  createdAt INTEGER NOT NULL
);

-- 2. 话题表
CREATE TABLE topics (
  id TEXT PRIMARY KEY,           -- 话题唯一ID
  title TEXT NOT NULL,           -- 话题标题
  hostIds TEXT NOT NULL,         -- 关联主机ID（JSON 数组）
  selectedProviderId TEXT,       -- 选中的 AI Provider
  selectedModelId TEXT,          -- 选中的 AI Model
  lastMessageAt INTEGER NOT NULL,-- 最后消息时间
  createdAt INTEGER NOT NULL
);

-- 3. 消息表
CREATE TABLE messages (
  id TEXT PRIMARY KEY,
  topicId TEXT NOT NULL,
  role TEXT NOT NULL,            -- 'user' | 'assistant' | 'system' | 'tool'
  content TEXT NOT NULL,
  timestamp INTEGER NOT NULL,
  thought TEXT,                  -- 思考过程（Agent）
  toolCalls TEXT,                -- 工具调用（JSON）
  toolCallId TEXT,               -- 关联的工具调用ID
  name TEXT,                     -- 工具名称
  FOREIGN KEY (topicId) REFERENCES topics(id) ON DELETE CASCADE
);

-- 4. 终端会话表
CREATE TABLE terminal_sessions (
  id TEXT PRIMARY KEY,
  topicId TEXT NOT NULL,
  hostId TEXT NOT NULL,
  hostAlias TEXT NOT NULL,
  name TEXT,                     -- 终端名称
  status TEXT NOT NULL,          -- 'active' | 'streaming' | 'closed' | 'disconnected'
  shellType TEXT,                -- shell 类型（bash/zsh）
  shellIntegrationReady INTEGER, -- Shell 集成是否就绪
  createdAt INTEGER NOT NULL,
  closedAt INTEGER,
  FOREIGN KEY (topicId) REFERENCES topics(id) ON DELETE CASCADE
);

-- 5. 终端 I/O 表（核心）
CREATE TABLE terminal_io (
  id TEXT PRIMARY KEY,
  sessionId TEXT NOT NULL,       -- 关联终端会话
  topicId TEXT NOT NULL,
  hostId TEXT NOT NULL,
  type TEXT NOT NULL,            -- 'input' | 'output'
  source TEXT NOT NULL,          -- 'agent' | 'user' | 'system'
  content TEXT NOT NULL,         -- 内容（输入命令或输出）
  exitCode INTEGER,              -- 退出码
  durationMs INTEGER,            -- 执行时长
  relatedInputId TEXT,           -- 关联的输入ID
  isStreaming INTEGER,           -- 是否流式输出
  chunkIndex INTEGER,            -- 块索引
  isTruncated INTEGER,           -- 是否被截断
  cwd TEXT,                      -- 当前工作目录
  taskId TEXT,                   -- 关联任务ID
  stepId TEXT,                   -- 关联步骤ID
  timestamp INTEGER NOT NULL,
  FOREIGN KEY (sessionId) REFERENCES terminal_sessions(id) ON DELETE CASCADE
);

-- 6. 任务表
CREATE TABLE tasks (
  id TEXT PRIMARY KEY,
  topicId TEXT NOT NULL,
  title TEXT NOT NULL,
  goal TEXT NOT NULL,            -- 任务目标
  status TEXT NOT NULL,          -- 'pending' | 'running' | 'completed' | 'failed'
  summary TEXT,                  -- 任务总结
  selectedProviderId TEXT,
  selectedModelId TEXT,
  createdAt INTEGER NOT NULL,
  updatedAt INTEGER NOT NULL
);

-- 7. 任务步骤表
CREATE TABLE task_steps (
  id TEXT PRIMARY KEY,
  taskId TEXT NOT NULL,
  type TEXT NOT NULL,            -- 'plan' | 'command' | 'result' | 'approval' | 'final'
  status TEXT NOT NULL,
  hostId TEXT,
  title TEXT,
  content TEXT NOT NULL,
  rawOutput TEXT,                -- 原始输出
  metadata TEXT,                 -- 元数据（JSON）
  startedAt INTEGER,
  endedAt INTEGER,
  createdAt INTEGER NOT NULL
);

-- 8. 记忆表
CREATE TABLE memories (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,            -- 'habit' | 'host_fact' | 'experience'
  content TEXT NOT NULL,
  hostId TEXT,
  topicId TEXT,
  importance INTEGER DEFAULT 3,  -- 1-5
  timestamp INTEGER NOT NULL
);

-- 9. 命令模式表（信任度学习）
CREATE TABLE command_patterns (
  id TEXT PRIMARY KEY,
  hostId TEXT NOT NULL,
  commandPattern TEXT NOT NULL,  -- 命令模式（正则）
  approvalCount INTEGER DEFAULT 0,
  rejectionCount INTEGER DEFAULT 0,
  trustLevel TEXT DEFAULT 'untrusted', -- 'untrusted' | 'familiar' | 'trusted'
  lastSeen INTEGER NOT NULL
);
```

## 五、上下文组装机制

### 5.1 ContextAssembler

**文件**: `src/main/agent/context-assembler.ts`

**分层组装策略**:

```typescript
class ContextAssembler {
  assemble(): AssembledContext {
    const messages: ChatCompletionMessageParam[] = []
    let usedTokens = 0

    // 第 1 层: System Prompt (最高优先级)
    if (this.systemPrompt) {
      messages.push({ role: 'system', content: this.systemPrompt })
      usedTokens += estimateTokens(this.systemPrompt)
    }

    // 第 2 层: 终端上下文 (高优先级)
    if (this.layers.has('terminal_context')) {
      const terminalCtx = this.buildTerminalContext()
      messages.push({ role: 'system', content: terminalCtx })
      usedTokens += estimateTokens(terminalCtx)
    }

    // 第 3 层: 记忆召回 (中优先级)
    if (this.layers.has('memory_recall')) {
      const memories = MemoryManager.recallRelevantContext(...)
      if (memories) {
        messages.push({ role: 'system', content: memories })
        usedTokens += estimateTokens(memories)
      }
    }

    // 第 4 层: 历史消息 (低优先级，可能被截断)
    const historyBudget = this.budget.usable - usedTokens
    const truncatedHistory = this.truncateHistory(this.history, historyBudget)
    messages.push(...truncatedHistory)

    // 第 5 层: 当前轮次消息
    messages.push(...this.turnMessages)

    return { messages, budget: { used: usedTokens, ... } }
  }
}
```

### 5.2 终端上下文构建

**文件**: `src/main/terminal.ts` - `buildTerminalContext()`

```typescript
buildTerminalContext(topicId: string): string {
  const sessions = this.getSessionsByTopic(topicId)

  let context = '### 当前终端会话:\n'

  for (const session of sessions) {
    context += `\n[${session.name}] (${session.hostAlias})\n`
    context += `- 状态: ${session.status}\n`
    context += `- 当前目录: ${session.cwd || 'unknown'}\n`
    context += `- 最后命令: ${session.lastCommand || 'none'}\n`
    context += `- 锁定状态: ${session.isLocked ? `被 ${session.lockedBy} 锁定` : '未锁定'}\n`

    // 最近输出摘要（最后 500 字符）
    if (session.lastOutput) {
      context += `- 最近输出: ${session.lastOutput.slice(-500)}\n`
    }
  }

  return context
}
```

## 八、现有架构的优势与局限

### 8.1 优势

| 方面           | 说明                                          |
| -------------- | --------------------------------------------- |
| **架构清晰**   | 分层明确：Service → Runner → Executor → Tools |
| **扩展性强**   | ToolRegistry 模式便于添加新工具               |
| **并发支持**   | executeGrouped 实现同主机串行、不同主机并行   |
| **容错机制**   | DoomLoop 检测、自动压缩、会话锁定             |
| **数据完整**   | 完整的 I/O 记录、任务追踪、记忆系统           |
| **多主机支持** | 本地 + SSH 远程统一管理                       |

### 8.2 局限

| 方面                 | 说明                            | 影响                        |
| -------------------- | ------------------------------- | --------------------------- |
| **单 Agent 瓶颈**    | 所有终端由单一 AgentRunner 管理 | 复杂场景下上下文膨胀        |
| **终端状态追踪弱**   | 仅通过 terminal_io 反推状态     | Agent 容易"忘记"终端状态    |
| **长时间任务支持弱** | 无专门监控机制                  | nvidia-smi 等任务状态不透明 |
| **缺乏终端分组**     | 无场景/标签概念                 | 多终端场景管理困难          |
| **模糊指令处理**     | 依赖用户明确指定终端            | 用户体验不够智能            |
| **会话恢复粒度粗**   | 仅恢复对话，终端全新创建        | 上下文丢失                  |

## 十、数据删除策略调查

### 10.1 外键约束分析

```sql
-- 关键外键约束行为
ON DELETE CASCADE   → 级联删除关联数据
ON DELETE SET NULL  → 保留记录，外键设为 NULL
```

### 10.2 删除 Topic 时的影响

| 数据表                | 删除行为 | 是否保留 | 说明                      |
| --------------------- | -------- | -------- | ------------------------- |
| **topics**            | -        | ❌ 删除  | 话题本身被删除            |
| **messages**          | CASCADE  | ❌ 删除  | 所有对话消息丢失          |
| **terminal_sessions** | CASCADE  | ❌ 删除  | 终端会话记录丢失          |
| **terminal_io**       | CASCADE  | ❌ 删除  | 命令执行历史丢失          |
| **tasks**             | CASCADE  | ❌ 删除  | 任务记录丢失              |
| **task_steps**        | CASCADE  | ❌ 删除  | 任务步骤丢失              |
| **approvals**         | CASCADE  | ❌ 删除  | 授权记录丢失              |
| **artifacts**         | CASCADE  | ❌ 删除  | 产物记录丢失              |
| **memories**          | SET NULL | ✅ 保留  | 记忆保留但失去 topic 关联 |

### 10.3 删除 Terminal 时的影响

| 数据表                | 删除行为 | 是否保留 | 说明                    |
| --------------------- | -------- | -------- | ----------------------- |
| **terminal_sessions** | -        | ❌ 删除  | 终端元数据删除          |
| **terminal_io**       | CASCADE  | ❌ 删除  | 该终端所有 I/O 记录删除 |

### 10.4 删除 Host 时的影响

| 数据表                | 删除行为 | 是否保留 | 说明                       |
| --------------------- | -------- | -------- | -------------------------- |
| **terminal_sessions** | SET NULL | ✅ 保留  | 终端保留但 hostId 为空     |
| **terminal_io**       | SET NULL | ✅ 保留  | I/O 记录保留但 hostId 为空 |
| **memories**          | SET NULL | ✅ 保留  | 记忆保留但 hostId 为空     |

### 10.5 关键发现

**当前设计的问题：**

1. **Topic 删除 = 数据全删**
   - 除了 memories 表，其他所有相关数据都会被级联删除
   - 命令历史、对话记录、任务记录全部丢失

2. **Terminal 删除 = I/O 全删**
   - 删除终端会话时，该终端的所有命令执行记录被级联删除
   - 无法追溯"这个终端曾经执行过什么"

3. **缺乏软删除机制**
   - 没有 `deletedAt` 字段
   - 没有归档表
   - 删除即永久删除

### 10.6 设计原则冲突

**Topic 为大隔离 vs 数据保留需求：**

- **现状**：删除 Topic/Terminal 时级联删除所有数据
- **问题**：模型无法知道"这个 topic 经历过什么"
- **需求**：即使删除 Topic 中的主机或终端，操作记录、命令、对话应该保留

**需要优化方向：**

- 实现逻辑删除（软删除）
- 保留历史记录但标记为"已删除"
- 给模型搜索工具查看历史记录
