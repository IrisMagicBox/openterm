# 突发奇想：Agent 架构设计讨论全过程

## 一、最初的问题提出

**用户提出的场景：**

> openterm 是一个终端管理 Agent，可以管理多个终端，对其下发命令、获取结果并根据用户自然语言对其进行下一步命令执行

### 场景 1: 完全操作本机的多终端

- 终端 1: 打开 a 目录，使用 terminal-assistant 探索
- 终端 2: 打开 b 目录，使用 claude code 探索
- 终端 3: 打开新终端，时刻监测当前的资源利用率

### 场景 2: 多远程 host 操作

- 同一个 host 操作多个终端
- 不同终端干不同的事情
- 示例（推理场景）:
  - 终端 1: 持续查看 nvidia-smi 监测
  - 终端 2: 推理模型
  - 终端 3: 本机 localhost 请求 chat 接口测试对话

### 核心问题

1. **架构问题**: 单 Agent 还是多 Agent？
   - 方案 A: 一个主 Agent 管理所有终端
   - 方案 B: 每个终端一个 SubAgent，主 Agent 统一协调

2. **存储问题**: 终端命令、自然语言对话如何存储？

3. **记忆问题**: 软件关闭后重新打开，对话是否能继续保留终端执行记录？

4. **管理问题**: 是否需要多个 agent 管理每个终端？
   - 每新开一个终端就下放一个 subagent
   - 对话 agent 作为主 agent 统一管理并和用户交互

## 二、架构方案对比讨论

### 方案 A: 集中式单 Agent

```
用户消息 → AgentRunner
              │
              ├── 工具调用
              │     ├── manage_terminal (创建终端)
              │     ├── execute_command (在终端1执行)
              │     ├── execute_command (在终端2执行)
              │     └── execute_command (在终端3执行)
              │
              └── 返回结果
```

**优点：**

- 简单，已完全实现
- 状态集中管理
- 易于调试

**缺点：**

- 复杂度随终端数量增加
- Agent 需要同时跟踪多个终端状态
- 长对话后上下文膨胀

### 方案 B: 分布式多 Agent（终端级 SubAgent）

```
用户消息 → 对话 Agent（主）
            │
            ├── 终端1 SubAgent ← 管理终端1生命周期
            ├── 终端2 SubAgent ← 管理终端2生命周期
            └── 终端3 SubAgent ← 管理终端3生命周期
```

**优点：**

- 职责更清晰，每个终端独立管理
- 单个终端崩溃不影响其他终端
- 便于扩展到更多终端类型

**缺点：**

- 需要新增架构
- Agent 间通信复杂度增加
- 实现难度大

### 最终决策

**采用方案 A 的增强版**：单 AgentRunner + 复杂任务 SubAgent

原因：

1. OpenTerm 已有成熟架构，单 Agent 管理多终端已完全实现
2. 通过 `task` 工具已经可以实现子 Agent（explore/verify）
3. "每个终端一个 Agent"过于复杂，收益不明显
4. 现有架构能满足 80% 需求，可以通过增强工具系统实现类似效果

## 三、关键设计问题讨论

### Q1: 终端创建策略

**用户决策**: 终端自动创建

用户说"在 a 目录用 terminal-assistant 探索，在 b 目录用 claude code 探索，同时监测资源"：

- 系统自动创建 3 个终端，分别执行对应命令
- 可以给终端打标签（`#terminal-assistant`, `#claude`, `#monitor`）便于管理

### Q2: 监控类任务处理

**用户决策**: 简化到终端，和其他命令一样

- `nvidia-smi` 监控在独立终端持续运行
- Agent 定期读取输出（类似 `tail -f`）
- 像人一样"看"终端，不需要专门的监控 Agent

### Q3: 自然语言到命令的映射 + UI 交互

**用户决策**: 结合 UI 交互

- **模糊时询问**: 用户说"探索这个目录"，模型询问"在哪个终端执行？"
- **明确时执行**: 用户说"用 terminal-assistant 探索"，直接执行
- **UI 交互**: 鼠标点击终端 → 自动添加 `@terminal-xxx` 到对话框
- 支持 `@terminal-1` 或 `@终端名称` 的方式指定

### Q4: 会话恢复策略

**用户决策**: 对话完全恢复，终端全新创建，历史不渲染

- **对话**: 完全恢复历史对话记录
- **终端**: 非正常关闭后，重新打开时创建全新终端
- **历史渲染**: ❌ 不渲染（避免 clear 命令后的困惑）
- **模型知晓**: ✅ Agent 通过上下文知道历史执行情况

### Q5: 架构模式

**用户决策**: Agent + 复杂任务 SubAgent

- **主 Agent**: 单 AgentRunner 管理所有终端
- **SubAgent**: 仅用于复杂任务（explore/verify）
- **不采用**: "每个终端一个 Agent" 架构（过于复杂）

### Q6: 记忆与会话追踪

**用户决策**: 需要会话级别的针对性记忆

**问题**: 长对话时 Agent 忘记终端执行情况
**解决**: 设计记忆机制
**目标**: 避免某些终端一直执行而 Agent 忘记

## 四、存储设计演变

### 第一阶段：复杂方案（巡查表概念）

**设计想法**: 类似于工厂的设备巡查表

**新增表结构**:

```sql
-- 主机巡查表
CREATE TABLE host_inspections (
  hostId TEXT PRIMARY KEY,
  purpose TEXT,              -- 主机用途
  systemInfo TEXT,           -- 系统信息
  cpuUsage REAL,             -- CPU 使用率
  currentStatus TEXT,        -- 当前状态
  lastOperation TEXT,        -- 上次操作
  notes TEXT,                -- 备注
  updatedAt INTEGER
);

-- 终端巡查表
CREATE TABLE terminal_inspections (
  sessionId TEXT PRIMARY KEY,
  purpose TEXT,              -- 终端用途
  currentStatus TEXT,        -- 当前状态
  runningProcessName TEXT,   -- 运行中进程
  progressInfo TEXT,         -- 进度信息
  notes TEXT,                -- 备注
  updatedAt INTEGER
);
```

**问题**: 字段太多，过于复杂，限制 Agent 自由度

### 第二阶段：极简方案（一张白纸）

**核心思想**: 给 Agent 一张白纸，让它自由记录

> 就像一个人拿着笔记本，随时记录：
>
> - 这个主机是干什么的
> - 这个终端用来干什么
> - 现在在做什么
> - 有什么需要注意的

**极简设计**:

```sql
-- 只新增 2 个字段！
ALTER TABLE hosts ADD COLUMN agentNotes TEXT;
ALTER TABLE terminal_sessions ADD COLUMN agentNotes TEXT;
```

**Agent 自由记录示例**:

```
【终端备注】
- 用途：GPU监控
- 当前：运行 nvidia-smi 监控（已30分钟）
- 状态：GPU利用率 85%，温度正常
- 最后更新：2024-01-15 14:30
```

**优势**:

- 极简：只新增 2 个字段
- 灵活：Agent 自由决定记录什么、怎么记录
- 无约束：不预设结构，适应各种场景
- 易维护：没有复杂的表结构和同步逻辑

## 五、关键决策总结

| 问题     | 决策                                                    |
| -------- | ------------------------------------------------------- |
| 架构模式 | 单 Agent + 复杂任务 SubAgent（不采用终端级 SubAgent）   |
| 终端创建 | 自动创建                                                |
| 监控任务 | 简化到终端，Agent 定期查看                              |
| 模糊指令 | UI 交互 + 模型询问                                      |
| 会话恢复 | 对话恢复，终端全新，历史不渲染                          |
| 状态追踪 | 极简方案：agentNotes 自由文本                           |
| 存储设计 | 只新增 2 个字段（host.agentNotes, terminal.agentNotes） |

## 七、反思与启发

### 重要教训

**"越多的字段意味着越复杂"**

- 最初设计：20+ 个字段，2 个新表，复杂的监控组件
- 最终方案：2 个字段，Agent 自由记录
- 结果：更简单、更灵活、更易维护

### 设计哲学

1. **不要预设太多结构** - 让 Agent 自己决定如何记录
2. **简单胜于复杂** - 2 个字段能解决的事情，不要用 20 个字段
3. **Agent 比你想象的更聪明** - 给它自由度，它能做好记录

### 未来扩展

即使现在只给了 2 个字段，未来如果需要：

- 可以提取常用模式，增加辅助字段
- 可以添加定期自动更新机制
- 可以增加智能提醒功能

**但核心永远是**：给 Agent 自由度，不要过度设计。
